# PayCameroon Complete Class Diagram Description

This document provides a comprehensive, exhaustive description of every entity, attribute, method, and relationship depicted in the PayCameroon System Class Diagram. It serves as the data dictionary and architectural blueprint for the platform's backend models.

## 1. Core Entities

### User
The central entity of the system. It represents any human interacting with the platform, differentiated by their `role`.
*   **Attributes:**
    *   `paycamId` (String): A unique identifier for the user within the ecosystem (e.g., PC-12345).
    *   `name` (String): Full legal name or display name.
    *   `phone` (String): Unique primary contact and login identifier.
    *   `pin` (String): Encrypted 4-digit PIN for transaction authorization.
    *   `password` (String): Encrypted password for web dashboard login.
    *   `balance` (Number): The current e-money balance in the user's digital wallet.
    *   `role` (String): Defines permissions. Values include: `user`, `agent`, `merchant`, `admin`, `finance`, `compliance`, `support`.
    *   `businessName` (String): Applicable only if role is `merchant` or `agent`.
    *   `kycVerified` (Boolean): Flag determining if the user has passed compliance checks.
    *   `status` (String): Account state (`active`, `frozen`, `blocked`).
    *   `language` (String): User's preferred UI language.
    *   `createdAt` / `updatedAt` (Date): Timestamps for record lifecycle.
*   **Methods:**
    *   `login()`: Authenticates the user via phone/password or phone/PIN.
    *   `updateProfile()`: Modifies basic user details.
    *   `changePassword()` / `updateLanguage()`: Updates security and localization settings.

### Transaction
The core financial ledger record. Every movement of money generates a Transaction object.
*   **Attributes:**
    *   `txId` (String): Unique transaction hash/identifier.
    *   `type` (String): Categorizes the movement (`p2p`, `cash_in`, `cash_out`, `merchant_payment`, `deposit`, `withdrawal`, `float_transfer`, `service_payment`).
    *   `amount` (Number): The principal amount being transferred.
    *   `fee` (Number): System taxes or fees charged to the user.
    *   `commission` (Number): The cut awarded to an Agent or Merchant for facilitating the transaction.
    *   `status` (String): Current state (`pending`, `completed`, `failed`, `reversed`).
    *   `timestamp` (Date): Execution time.
    *   `description` (String): Human-readable summary of the transaction.
*   **Methods:**
    *   `processTransaction()`: Executes the ledger movement across wallets.
    *   `calculateFees()`: Fetches global rates from `SystemSettings` to compute `fee` and `commission`.
    *   `reverseTransaction()`: Handles refunds in case of failure.

### KYCApplication
Holds data for Know Your Customer compliance checks.
*   **Attributes:**
    *   `documentType` (String): e.g., "National ID", "Passport".
    *   `documentUrl` / `selfieUrl` (String): Storage pointers to the uploaded images.
    *   `status` (String): `pending`, `approved`, or `rejected`.
    *   `comments` (String): Notes left by the Compliance Officer.
    *   `submittedAt` / `reviewedAt` (Date): Lifecycle timestamps.
*   **Methods:**
    *   `approve()` / `reject()`: Transitions state and updates the parent `User.kycVerified` flag.

### AMLAlert
Threat detection flags generated by the AI heuristic engine.
*   **Attributes:**
    *   `alertId` (String): Unique ID for the alert.
    *   `ruleTriggered` (String): The AI rule broken (e.g., "Velocity Anomaly", "Location Spoofing").
    *   `severity` (String): Impact level (`Low`, `Medium`, `High`, `Critical`).
    *   `description` (String): AI-generated explanation of the threat.
    *   `status` (String): Investigation lifecycle (`open`, `investigating`, `resolved`).
*   **Methods:**
    *   `investigate()`: Marks alert as under review by Compliance.
    *   `resolve()`: Closes the alert.

### ApprovalRequest
A workflow entity for requests that require manual Finance Admin authorization.
*   **Attributes:**
    *   `requestId` (String): Unique workflow ID.
    *   `type` (String): Either `float_request` (Agent) or `merchant_withdrawal` (Merchant).
    *   `amount` (Number): Value requested.
    *   `bankDetails` (String): Destination details for withdrawals, or proof-of-deposit for float requests.
    *   `status` (String): `pending`, `approved`, or `rejected`.
*   **Methods:**
    *   `approveRequest()`: Triggers money movement (draws from `InternalWallet`).
    *   `rejectRequest()`: Denies the request.

### SupportTicket & ChatMessage (PayChat System)
Manages the customer service messaging architecture.
*   **SupportTicket Attributes:** `ticketId`, `subject`, `status` (`open`, `closed`).
*   **ChatMessage Attributes:** `messageId`, `content`, `isRead`, `timestamp`.
*   **Methods:** `addMessage()`, `closeTicket()`.

### Notification
Push/In-app alerts delivered to users.
*   **Attributes:** `notificationId`, `title`, `message`, `type`, `isRead`.
*   **Methods:** `markAsRead()`.

### SystemSettings
A global singleton-like configuration model managed by the Super Admin.
*   **Attributes:**
    *   `logoUrl` (String): Platform branding.
    *   `taxRate`, `p2pFee`, `cashOutFee` (Number): Global transaction fees.
    *   `agentCommissionRate`, `merchantCommissionRate` (Number): Revenue share metrics.
    *   `securityPolicies` (Object): Active AML thresholds and system limits.
*   **Methods:**
    *   `updateSettings()`: Saves new global variables.
    *   `getRates()`: Used by `Transaction` logic.

### AuditLog
Immutable security logs.
*   **Attributes:** `logId`, `action` (e.g., "User Blocked"), `category`, `ipAddress`, `details`, `timestamp`.

### InternalWallet
Platform-owned accounts representing the system's corporate treasury.
*   **Attributes:**
    *   `walletId` (String): Wallet identifier.
    *   `name` (String): e.g., "Treasury", "Revenue Wallet", "Commission Wallet".
    *   `balance` (Number): Total funds held.
    *   `type` (String): Categorization of funds.
*   **Methods:** `credit()`, `debit()`.

## 2. Relationships & Multiplicities

### User-Centric Relationships
*   **User to Transaction (1 to 0..*)**: A User can initiate many transactions (as Sender) and receive many transactions (as Recipient).
*   **User to KYCApplication (1 to 0..1)**: A Standard User submits up to one active KYC application at a time.
*   **User to SupportTicket (1 to 0..*)**: A User (or Agent/Merchant) can open multiple support tickets.
*   **User to Notification (1 to 0..*)**: A User receives multiple system notifications.
*   **User to ApprovalRequest (1 to 0..*)**: Agents and Merchants generate multiple workflow requests over time.

### Role-Based (Administrative) Relationships
*   **Compliance Admin to KYCApplication / AMLAlert**: Compliance Officers read, investigate, and resolve these entities.
*   **Support Admin to SupportTicket**: Support Reps are assigned to and resolve tickets.
*   **Finance Admin to ApprovalRequest**: Finance admins review, approve, or reject these operational requests.
*   **Super Admin to SystemSettings / AuditLog**: Super Admins mutate the settings and generate audit logs.

### System Object Relationships
*   **SupportTicket to ChatMessage (1 to 1..* - Composition)**: A Support Ticket is comprised of one or more Chat Messages. If the ticket is deleted, messages are typically archived or deleted.
*   **ChatMessage to User (0..* to 1)**: Every message is authored by a User (either the customer or the Support Rep).
*   **Transaction to SystemSettings (0..* to 1)**: Every financial transaction dynamically queries the `SystemSettings` to calculate current applicable taxes and commissions at the exact time of execution.
*   **ApprovalRequest to InternalWallet**: When a `float_request` is approved, the system automatically debits the central `InternalWallet` (Treasury) to credit the Agent's User wallet. When transaction fees are collected, the system credits the `InternalWallet` (Revenue).
